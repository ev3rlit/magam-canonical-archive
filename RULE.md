이 프로젝트의 TypeScript는 Go의 철학을 따른다. 목표는 문법을 흉내내는 것이 아니라, 읽기 쉽고 예측 가능하며 유지보수 비용이 낮은 구조를 만드는 것이다.

## 핵심 원칙

- 코드는 한 번에 이해되어야 한다.
- 입력, 출력, 부작용, 실패 경로는 코드에 드러나야 한다.
- 숨겨진 상태 변경, 암묵적 기본값, 매직한 후처리보다 명시적 전달을 우선한다.
- 함수, 타입, 모듈은 한 가지 책임만 가진다.
- 짧은 코드보다 명확한 코드를 선택한다.
- 작은 중복은 성급한 공통화보다 낫다.

## 인터페이스

- 인터페이스는 구현이 아니라 필요한 능력만 표현한다.
- 가능한 경우 1~3개의 메서드만 가진다.
- 읽기와 쓰기, 조회와 변경, 계산과 동기화를 분리한다.
- 인터페이스는 제공자보다 소비자 가까이에 둔다.
- 하나의 구현체만 있고 실제 대체 가능성이 없으면 인터페이스를 만들지 않는다.
- 입력은 좁은 인터페이스로 받고, 출력은 구체 타입으로 돌려준다.
- 인터페이스 이름은 역할이 드러나야 한다.
- 이름은 가능하면 도메인 명사 + 역할로 짓는다. 예: `CanvasStore`, `ObjectLoader`, `PatchWriter`, `LayoutEngine`
- 범용 경계에서는 역할 자체가 보이는 이름을 쓴다. 예: `Reader`, `Writer`, `Clock`, `Parser`

## 타입

- 타입은 런타임 동작을 설명해야지 감추면 안 된다.
- 복잡한 generic, conditional type, mapped type, infer 체인은 마지막 수단이다.
- 상태 표현에는 여러 boolean보다 discriminated union을 우선한다.
- 데이터 모양에는 단순한 object type 또는 type alias를 우선한다.
- 확장 가능한 계약이 실제로 필요할 때만 interface를 쓴다.
- `as any`, `as unknown as` 같은 단언 중심 설계를 피한다.

## 함수와 모듈

- 함수는 한 가지 일만 한다.
- 함수 이름은 효과를 바로 드러내야 한다.
- 함수 시그니처는 작고 분명해야 한다.
- 인자가 적고 의미가 분명하면 options object로 감싸지 않는다.
- 모듈은 하나의 개념 또는 하나의 작은 협력 단위만 담는다.
- public surface는 작게 유지하고 내부 세부사항은 export하지 않는다.
- 공유 가능한 변경 상태보다 입력과 반환값 중심의 흐름을 선호한다.

## 의존성과 추상화

- 의존성은 생성 시점, 팩토리, 함수 인자로 명시적으로 드러낸다.
- 전역 singleton, 숨겨진 context, import만으로 이어지는 결합을 피한다.
- 상속보다 조합을 우선한다.
- 추상화는 현재 비용을 줄일 때만 도입한다.
- 미래 확장성만을 이유로 계층, 래퍼, 추상 계층을 만들지 않는다.

## 에러와 경계

- 실패는 명시적으로 드러낸다.
- 로그만 남기고 삼키는 에러, 성공처럼 보이는 fallback, 조용한 early return을 피한다.
- 외부 입력과 불안정한 데이터는 경계에서 검증한다.
- 내부 로직은 검증된 데이터 위에서 단순하게 동작해야 한다.
- 에러 메시지는 무엇이, 어디서, 왜 실패했는지 드러내야 한다.

## 기본적으로 피할 것

- 미래 확장성만을 이유로 한 인터페이스 도입
- 하나의 구현체를 감싸는 무의미한 추상화
- 과한 generic과 타입 체조
- 넓은 options object와 숨겨진 기본값
- 호출부를 단순하게 보이게 하려고 내부 복잡성을 숨기는 설계
- 읽는 사람의 추론에 의존하는 축약형 코드
