# 기능 명세: 조밀한 MindMap 레이아웃

**기능 브랜치**: `005-mindmap-layout`  
**작성일**: 2026-03-11  
**상태**: 초안  
**입력**: 사용자 설명: "겹치지 않으면서 사람이 손으로 정리한 것처럼 조밀하게 배치되는 신규 MindMap 레이아웃을 만든다. 이 레이아웃은 형제 노드의 과도한 수직 확장을 줄이고, 깊은 서브트리 때문에 생기는 불필요한 공백을 완화하며, 사용자가 작성한 MindMap topology를 그대로 유지해야 한다. 또한 여러 레이아웃 모드를 전환하기보다 하나의 일관된 compact 규칙으로 동작해야 한다."

## 사용자 시나리오 및 테스트 *(mandatory)*

### 사용자 스토리 1 - 조밀하고 읽기 쉬운 마인드맵 (우선순위: P1)

다이어그램 작성자는 신규 MindMap 레이아웃을 선택해, 가지들이 서로 겹치지 않으면서도 조밀하게 배치된 결과를 얻을 수 있어야 한다. 결과는 기계적으로 벌어진 트리보다 사람이 손으로 정리한 마인드맵에 더 가까워야 한다.

**우선순위 이유**: 이 기능의 핵심 가치는 더 조밀하고 더 읽기 쉬운 마인드맵 배치다. 이 요구를 만족하지 못하면 개선 목적 자체를 달성하지 못한다.

**독립 검증 방법**: 승인된 벤치마크 마인드맵 샘플을 신규 레이아웃으로 렌더링해, 노드 겹침이 없고 동일한 정보가 기존 기준 레이아웃보다 더 작은 캔버스 면적 안에 배치되는지 확인하면 독립적으로 검증할 수 있다.

**수용 시나리오**:

1. **Given** 크기가 서로 다른 노드를 포함한 MindMap이 있을 때, **When** 작성자가 신규 레이아웃을 적용하면, **Then** 최종 배치에는 노드 간 겹침이 없어야 하고 기존 부모-자식 관계가 그대로 유지되어야 한다.
2. **Given** 동일한 MindMap을 기존 레이아웃과 신규 레이아웃으로 각각 렌더링했을 때, **When** 작성자가 결과를 비교하면, **Then** 신규 레이아웃은 읽기 쉬움을 유지하면서 더 작은 전체 캔버스 면적을 사용해야 한다.
3. **Given** 내용과 설정이 바뀌지 않은 같은 MindMap이 반복 렌더링될 때, **When** 레이아웃을 다시 계산하면, **Then** 노드 배치는 일관되고 예측 가능해야 한다.
4. **Given** 하나의 MindMap 그룹 안에 여러 루트 노드가 있을 때, **When** 신규 레이아웃을 적용하면, **Then** 각 루트 서브트리는 개별적으로 조밀하게 배치되고 루트들 사이도 같은 compact 규칙으로 정리되어야 한다.

---

### 사용자 스토리 2 - 형제가 많은 가지도 조밀하게 유지됨 (우선순위: P2)

다이어그램 작성자는 한 부모에 자식이 많아도 가지가 하나의 긴 수직 열처럼 무너져 내리지 않는 결과를 얻을 수 있어야 한다. 형제가 많은 마인드맵도 여전히 조밀하고 빠르게 훑어볼 수 있어야 한다.

**우선순위 이유**: 형제 노드가 많을 때 수직으로 과도하게 늘어나는 문제는 현재 레이아웃의 핵심 한계 중 하나다.

**독립 검증 방법**: 형제 수가 많은 벤치마크 마인드맵을 렌더링해, 기존 기준 레이아웃 대비 가지 높이와 주변 공백이 줄어드는지 확인하면 독립적으로 검증할 수 있다.

**수용 시나리오**:

1. **Given** 직접 자식이 많은 부모 노드가 있을 때, **When** 신규 레이아웃을 적용하면, **Then** 해당 가지는 더 이상 하나의 매우 긴 수직 열을 중심으로 커지지 않아야 한다.
2. **Given** 폭과 높이가 서로 다른 형제 서브트리들이 있을 때, **When** 신규 레이아웃을 적용하면, **Then** 형제들은 균일하게 떨어진 행(row)처럼 보이기보다 하나의 시각적 덩어리로 읽혀야 한다.
3. **Given** 얕지만 넓은 자식 가지들이 함께 있는 부모가 있을 때, **When** 레이아웃을 다시 계산하면, **Then** 형제 영역은 눈에 띄는 다른 모드 전환 없이 자연스럽게 확장되어야 한다.

---

### 사용자 스토리 3 - 깊은 서브트리가 주변 공간을 낭비하지 않음 (우선순위: P3)

다이어그램 작성자와 조회자는 깊은 가지와 얕은 가지가 함께 있는 마인드맵을 볼 때, 특정 깊은 서브트리 하나 때문에 주변 가지 전체가 과도하게 벌어지지 않는 결과를 얻을 수 있어야 한다.

**우선순위 이유**: 깊은 서브트리 때문에 형제 간격과 수평 공백이 함께 커지는 문제는 현재 레이아웃의 또 다른 핵심 결함이다.

**독립 검증 방법**: 깊은 가지와 얕은 가지가 섞인 벤치마크 마인드맵, 그리고 늦게 크기가 확정되는 콘텐츠를 포함한 샘플을 렌더링해, 인접 가지 간 공백이 줄고 최종 상태가 안정적으로 수렴하는지 확인하면 독립적으로 검증할 수 있다.

**수용 시나리오**:

1. **Given** 한 형제 서브트리가 다른 형제들보다 훨씬 더 깊을 때, **When** 신규 레이아웃을 적용하면, **Then** 인접한 얕은 형제들은 해당 깊은 서브트리가 만든 전체 빈 공간을 그대로 떠안지 않고 가깝게 유지되어야 한다.
2. **Given** 노드 콘텐츠가 렌더링 완료 후 더 커지는 MindMap이 있을 때, **When** 레이아웃이 갱신되면, **Then** 최종 상태는 겹침 없는 안정된 배치로 수렴해야 한다.
3. **Given** 하나의 캔버스 안에 여러 MindMap이 있을 때, **When** 한 그룹의 레이아웃이 계산되거나 갱신되면, **Then** 다른 그룹의 내부 레이아웃 동작은 영향을 받지 않아야 한다.

## 엣지 케이스

- 부모가 자식을 하나만 가지는 경우에도 불필요한 확산이 생기지 않아야 한다.
- 직접 자식 수가 매우 많은데 각 라벨은 짧은 경우를 처리할 수 있어야 한다.
- 한 형제 서브트리는 매우 깊고, 인접 형제들은 얕지만 넓은 경우를 처리할 수 있어야 한다.
- 한 형제 서브트리만 다른 형제들보다 훨씬 큰 노드들을 포함하는 경우를 처리할 수 있어야 한다.
- 하나의 MindMap 그룹 안에 여러 루트 노드가 공존하는 경우에도 각 루트 서브트리와 루트 간 배치가 모두 compact해야 한다.
- 리치 콘텐츠가 초기 렌더 이후 노드 크기를 바꾸는 경우를 처리할 수 있어야 한다.
- 여러 MindMap이 같은 세션에서 함께 배치되는 경우를 처리할 수 있어야 한다.
- 작은 내용 수정이 전체 맵의 과도한 점프로 이어지지 않아야 한다.

## 가정

- 이번 단계에서는 기존 레이아웃을 대체하지 않고, 새로운 opt-in MindMap 레이아웃을 추가한다.
- 레이아웃은 사용자가 작성한 MindMap topology를 정확히 그대로 유지해야 한다.
- 벤치마크 평가는 형제 과다, 깊은 가지, 혼합 크기, 늦은 크기 확정 사례를 포함한 승인된 fixture 세트를 기준으로 진행한다.
- 시각 품질 비교는 현재 `tree`, `bidirectional` 레이아웃을 기준선으로 삼는다.

## 의존성

- 비교 및 회귀 검증을 위한 공용 벤치마크 fixture 세트가 정의되어 있어야 한다.
- 기존 레이아웃 기준선 결과를 반복 가능하게 비교할 수 있어야 한다.

## 요구사항 *(mandatory)*

### 기능 요구사항

- **FR-001**: 시스템은 신규 레이아웃을 선택 가능한 MindMap 레이아웃 옵션으로 제공해야 한다.
- **FR-002**: 시스템은 사용자가 작성한 MindMap topology를 그대로 보존해야 하며, 노드를 재그룹화하거나 부모-자식 관계를 바꾸거나 가상 계층을 삽입해서는 안 된다.
- **FR-003**: 시스템은 최종 렌더 상태에서 노드 간 겹침이 없도록 노드를 배치해야 한다.
- **FR-004**: 시스템은 넓은 가지를 별도의 눈에 띄는 레이아웃 모드로 전환하지 않고, 트리 전체에 하나의 일관된 배치 규칙을 사용해야 한다.
- **FR-005**: 시스템은 한 부모에 직접 자식이 많을 때 과도한 수직 확장을 줄여야 한다.
- **FR-006**: 시스템은 형제 가지들이 서로 고립된 행처럼 보이지 않고 하나의 그룹으로 읽히도록 시각적으로 묶어야 한다.
- **FR-007**: 시스템은 하나의 깊은 형제 서브트리 때문에 인접한 얕은 형제들 주변에 동일한 규모의 빈 공간이 강제되지 않도록 해야 한다.
- **FR-008**: 시스템은 혼합된 노드 크기를 지원하면서도 전체 레이아웃 밀도와 가지 가독성을 유지해야 한다.
- **FR-009**: 시스템은 동일한 topology, 콘텐츠, 레이아웃 설정에 대해 결정적인 결과를 유지해야 한다.
- **FR-010**: 시스템은 맵이 점진적으로 바뀔 때 영향받은 가지 바깥의 불필요한 이동을 최소화해야 한다.
- **FR-011**: 시스템은 렌더 후 의미 있는 크기 변화가 발생한 뒤에도 안정된 최종 배치에 도달해야 한다.
- **FR-012**: 시스템은 여러 MindMap 그룹을 서로 독립적으로 유지하여, 한 그룹의 레이아웃 갱신이 다른 그룹의 내부 구조를 바꾸지 않도록 해야 한다.
- **FR-013**: 시스템은 특정 맵에서 완벽하게 조밀한 배치를 만들 수 없더라도, 여전히 읽기 쉽고 사용 가능한 결과를 유지해야 한다.

### 핵심 엔티티 *(include if feature involves data)*

- **MindMap 레이아웃 모드**: 하나의 MindMap 그룹에 적용되는 선택 가능한 배치 방식이다.
- **MindMap topology**: 레이아웃이 정확히 보존해야 하는 사용자 작성 부모-자식 구조이며, 하나 이상의 루트 노드를 포함할 수 있다.
- **가지 클러스터**: 하나의 부모 주변에서 형제 서브트리들이 만들어내는 시각적 묶음이다.
- **최종 렌더 노드 점유 영역**: 노드 콘텐츠 렌더링이 완료된 뒤 실제로 차지하는 시각적 영역이다.
- **벤치마크 fixture 세트**: 레이아웃 품질 비교와 회귀 방지를 위해 사용하는 대표 마인드맵 샘플 모음이다.

## 성공 기준 *(mandatory)*

### 측정 가능한 결과

- **SC-001**: 승인된 벤치마크 fixture 세트에서 최종 렌더 상태의 노드 겹침은 100% 없어야 한다.
- **SC-002**: 승인된 벤치마크 fixture 세트에서 신규 레이아웃은 현재 `tree` 기준선보다 전체 캔버스 면적을 최소 20% 줄이고, 현재 `bidirectional` 기준선보다 최소 10% 줄여야 한다.
- **SC-003**: 형제 수가 많은 벤치마크 맵에서 가장 큰 형제 클러스터의 최대 수직 span은 현재 `tree` 기준선 대비 최소 30% 감소해야 한다.
- **SC-004**: 깊은 형제 서브트리와 얕은 형제 서브트리가 함께 있는 벤치마크 맵에서 인접 형제 가지 사이 평균 수평 공백은 현재 `tree` 기준선 대비 최소 25% 감소해야 한다.
- **SC-005**: 렌더 후 크기 변화가 있는 벤치마크 맵에서, 콘텐츠 크기 변화가 멈춘 뒤 두 번의 자동 보정 사이클 이내에 겹침 없는 안정된 최종 상태에 도달해야 한다.
- **SC-006**: 반복 렌더 회귀 fixture에서 동일한 입력과 설정은 실행마다 1px 이내 오차 범위의 동등한 노드 위치를 만들어야 한다.
