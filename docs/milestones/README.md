# Magam - Context Document

## 문서 목적

이 문서는 새로운 세션에서 제품 컨텍스트를 빠르게 공유하기 위한 기준 문서이다.
대상 제품은 **Miro, FigJam, Freeform, Excalidraw와 유사한 로컬 기반 데스크톱 캔버스 앱**이며, 핵심 차별점은 **사용자가 이미 보유한 외부 AI 도구와 협업 편집이 가능하도록 설계된 구조**에 있다.

이 문서는 다음 내용을 정리한다.

- 제품 정체성
- 핵심 설계 원칙
- 사용자 목표
- 정보 구조
- 주요 화면 목록
- 주요 사용자 행위
- 외부 AI 협업 개념
- 설계 시 주의할 점

---

## 1. 제품 개요

### 한 줄 정의

로컬 파일 시스템 기반의 데스크톱 캔버스 앱이며, 사용자가 직접 편집할 수 있을 뿐 아니라 **외부 AI 도구가 사용자와 함께 캔버스를 편집할 수 있도록 설계된 협업형 편집 환경**이다.

### 유사 제품군

- Miro
- FigJam
- Apple Freeform
- Excalidraw

### 차이점

이 앱은 SaaS 중심 협업 서비스가 아니라, **사용자 로컬 파일 시스템을 기반으로 동작하는 데스크톱 앱**이다.
또한 앱이 자체적으로 AI 모델이나 AI 채팅 기능을 제공하는 것이 아니라, **사용자가 이미 보유한 AI 도구를 활용하여 캔버스를 편집**하도록 돕는다.

---

## 2. 핵심 제품 철학

### 우선순위

1. **Mobile First**
2. **AI-native collaboration**

### Mobile First의 의미

- 데스크톱 앱이어도 모바일 우선 UX 원칙으로 화면 구조를 설계한다.
- 복잡한 좌우 패널 고정 구조보다, 전환 가능한 패널, 하단 시트, 압축된 툴바 중심 구조를 우선 고려한다.
- 핵심 작업 흐름은 좁은 화면에서도 수행 가능해야 한다.

### AI-native collaboration의 의미

- 앱 내부에 AI를 탑재하는 것이 목표가 아니다.
- 사용자가 이미 갖고 있는 AI 도구가 이 앱의 캔버스를 다룰 수 있도록 설계하는 것이 목표다.
- 기존의 “사람 ↔ 사람 협업” 개념을 “사람 ↔ AI 협업”으로 확장한다.

---

## 3. 이 앱이 제공하는 것 / 제공하지 않는 것

### 앱이 직접 제공하는 것

- 로컬 워크스페이스 관리
- 캔버스 생성 / 삭제 / 편집
- 템플릿 관리
- 컴포넌트 관리
- 로컬 파일 시스템 기반 저장
- 외부 도구가 접근 가능한 편집 환경
- 변경 검토 / 승인 / 복구 / 충돌 해결
- 활동 기록 및 버전 히스토리

### 앱이 직접 제공하지 않는 것

- 자체 AI 모델
- 자체 AI API
- 자체 AI 프록시 서버
- 자체 채팅 SaaS
- 자체 LLM 과금 / 라우팅 시스템

즉, 이 앱은 **AI를 제공하는 앱**이 아니라, **AI가 다룰 수 있는 앱**이다.

---

## 4. 외부 AI 협업 개념

### 기본 개념

사용자는 이미 보유한 AI 도구를 사용한다. 예:

- Claude Code
- Codex
- Antigravity
- OpenCode
- 기타 사용자 보유 도구

이 도구들은 사용자의 구독, API, 로컬 실행 환경을 사용하며, 본 앱은 이를 대체하지 않는다.

### 협업 모델

기존 협업 모델:

- 사용자 ↔ 사용자

확장 협업 모델:

- 사용자 ↔ AI
- 사용자 ↔ 여러 AI 도구
- 사용자 ↔ AI ↔ 캔버스 오브젝트

### 중요한 관점

AI는 앱 내부 기능이 아니라, **외부 협업 편집 주체(actor)** 로 본다.

따라서 제품 구조에서 중요한 것은 다음이다.

- 연결
- 권한 범위
- 변경 승인
- 활동 로그
- 충돌 해결
- 복구 가능성

---

## 5. 주요 사용자 목표

사용자는 이 앱에서 다음을 원한다.

1. 로컬 폴더를 워크스페이스로 등록하고 관리한다.
2. 워크스페이스 안에서 여러 캔버스를 만든다.
3. 캔버스에 오브젝트를 직접 편집한다.
4. 템플릿과 컴포넌트를 재사용한다.
5. 자신이 가진 외부 AI 도구를 이용해 캔버스를 편집한다.
6. 외부 도구가 만든 변경을 검토하고 승인하거나 되돌린다.
7. 사람 편집과 AI 편집이 섞이는 상황에서도 안전하게 작업한다.

사이드바 기준 1차 메뉴 제안:

- 워크스페이스
- 최근 항목
- 템플릿
- 컴포넌트
- 연결 / 협업
- 설정

### 메뉴 해석

#### 워크스페이스

로컬 폴더 기반 작업 공간 관리의 중심 메뉴.

#### 최근 항목

최근 열었던 캔버스, 최근 편집한 워크스페이스, 재진입을 위한 빠른 접근 메뉴.

#### 템플릿

캔버스 템플릿 관리 메뉴.

#### 컴포넌트

오브젝트 템플릿, 차트, 표, 플러그인 블록 등 재사용 단위 관리 메뉴.

#### 연결 / 협업

외부 도구가 앱과 어떻게 상호작용하는지 관리하는 메뉴.
AI 전용 메뉴가 아니라 **외부 편집 주체 관리 메뉴**이다.

#### 설정

앱 전역 환경 설정, 저장 정책, 권한 정책, 외부 편집 기본 정책 등을 관리하는 메뉴.

## 10. 캔버스 편집기에서 중요한 개념

캔버스 편집기는 단순히 오브젝트를 그리는 화면이 아니라, **사용자 편집과 외부 도구 편집이 동시에 반영될 수 있는 편집 공간**이어야 한다.

따라서 다음 상태를 지원해야 한다.

- 현재 외부 도구가 이 캔버스를 관찰 중
- 외부 편집 요청 대기 중
- 변경 승인 필요
- 자동 적용 중지
- 충돌 발생
- 복구 가능 상태

중요한 점은, 편집기 안에 별도 “AI 채팅”이 중심이 아니라, **협업 상태 레이어**가 자연스럽게 녹아 있어야 한다는 점이다.

## 11. 설계 시 주의할 점

### 11.1 AI를 내장 기능처럼 보이게 만들지 말 것

잘못된 방향:

- AI 메뉴
- AI 챗 탭
- 앱 내부 AI 비서처럼 보이는 구조

권장 방향:

- 연결
- 협업
- 외부 편집 주체
- 변경 승인
- 활동 로그

### 11.2 제품의 핵심은 “연동”보다 “편집 안전성”이다

사용자는 외부 AI가 캔버스를 바꿀 수 있다는 사실보다, 다음을 더 중요하게 생각한다.

- 어디까지 바꿀 수 있는가
- 내가 승인해야 하는가
- 잘못 바뀌면 되돌릴 수 있는가
- 누가 바꿨는지 알 수 있는가

### 11.3 로컬 파일 기반의 현실적인 문제를 반드시 고려할 것

- 권한 거부
- 경로 이동
- 디렉토리 접근 실패
- 외부 프로그램이 파일을 변경한 경우
- 충돌 / 손상 / 덮어쓰기 문제

### 11.4 Mobile First는 단순 축소판이 아니다

데스크톱 레이아웃을 줄이는 방식이 아니라,

- 패널을 하단 시트로 전환
- 도구를 압축된 액션바로 제공
- 보조 기능을 화면 전환형 구조로 설계

해야 한다.
