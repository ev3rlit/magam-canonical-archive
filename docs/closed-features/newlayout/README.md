# New MindMap Layout

## 배경

현재 `tree`, `bidirectional` 레이아웃은 기존 구조보다 개선되었지만, 여전히 다음 문제가 남아 있습니다.

- 노드 크기가 제각각인 상황에서 노드 간 충돌이 발생할 수 있다.
- 형제 노드가 한 방향, 특히 수직으로 길게 늘어나면서 빈 공간이 많고 전체 밀도가 낮다.
- 배치가 기계적으로 보여, 사람이 직접 정리한 마인드맵 같은 응집감이 부족하다.
- 형제 수가 많아질수록 2차원적으로 퍼뜨리기보다 선형으로 누적되어 캔버스 높이만 커지는 경향이 있다.
- 특정 형제의 자식 트리가 깊어질수록 형제 노드들 사이 간격이 과도하게 벌어지고, 그 여파로 수평 방향 dead space도 함께 커진다.

이번 개선의 전제는 명확합니다.

- 마인드맵은 더 촘촘하게 배치되어야 한다.
- 어떤 노드도 겹치면 안 된다.
- 결과는 "자동 정렬된 트리"보다 "사람이 읽기 좋게 손으로 정리한 마인드맵"에 가까워야 한다.

## 문제 정의

### 사용자 관점

1. 구조가 간단해도 화면을 불필요하게 넓게 사용해 한눈에 보기 어렵다.
2. 관련된 가지(branch)가 시각적으로 뭉치지 않아 읽는 흐름이 끊긴다.
3. 노드가 가까워 보이더라도 실제로는 충돌하거나 엣지가 어색하게 겹쳐 보일 수 있다.
4. 형제 노드가 많아지는 순간 특정 축으로만 길어져, 사람이 정리한 마인드맵보다 훨씬 둔하고 답답하게 보인다.
5. 한 형제의 서브트리만 깊어져도 주변 형제 전체가 같이 벌어져, 실제 정보량보다 훨씬 많은 빈 공간이 생긴다.

### 제품 관점

1. 현재 레이아웃은 "겹치지 않게 정렬"보다 "방향성 유지"에 더 가깝다.
2. 다양한 크기의 노드가 섞일수록 공간 활용도가 급격히 떨어진다.
3. 향후 polymorphic children, rich content 노드가 늘어날수록 현재 방식의 한계가 더 커진다.
4. 형제 노드 증가를 타일형, 방사형, 혹은 덩어리 재구성으로 흡수하지 못하고 단순 나열로 처리한다.
5. 깊은 서브트리의 세로 높이가 형제 간격 계산에 그대로 전파되어, 결과적으로 가로 영역까지 비효율적으로 점유한다.

### 사람이 그리는 마인드맵과의 차이

사람은 형제 노드가 많아질 때 단순히 한 줄로 계속 늘어놓지 않습니다.

1. 수직으로 너무 길어질 것 같으면 타일맵처럼 여러 덩어리로 나눠 배치한다.
2. 중심 주제 주변을 방사형 또는 부채꼴에 가깝게 퍼뜨려 시선을 분산시킨다.
3. 필요하면 모든 항목을 같은 깊이의 형제로 두지 않고, 사용자가 직접 더 깊은 트리 구조로 다시 정리하기도 한다.

다만 이 마지막 결정은 레이아웃이 아니라 작성(authoring) 단계의 몫입니다.
신규 레이아웃은 사용자가 정한 트리 구조를 바꾸지 않고, 그 구조를 더 읽기 좋고 더 조밀하게 배치하는 데 집중합니다.

## 목표

### 핵심 목표

1. 사람이 그린 것처럼 오밀조밀한 마인드맵 배치를 만든다.
2. 노드와 노드, 노드와 엣지가 시각적으로 충돌하지 않도록 한다.
3. 빈 공간을 줄이되, 읽기 어려운 과밀 상태는 만들지 않는다.
4. 부모-자식, 형제-형제 관계가 자연스럽게 보이도록 가지별 응집도를 높인다.
5. 동일한 입력에서는 항상 예측 가능한 배치 결과를 만든다.

### 품질 목표

1. 기존 `tree`, `bidirectional` 대비 더 작은 화면 면적 안에 동일한 정보를 배치한다.
2. 레이아웃 결과가 한두 개 노드 추가만으로 전체를 과도하게 흔들지 않도록 한다.
3. 큰 노드와 작은 노드가 섞여도 전체 균형이 무너지지 않도록 한다.

## 핵심 경험 원칙

### 1) Compact First

- 서브트리 사이의 불필요한 간격을 줄인다.
- 같은 깊이보다 같은 의미 단위를 우선해서 묶어 보이게 한다.

### 2) Collision Free

- 최종 결과에서 노드 박스는 절대 겹치지 않아야 한다.
- 비동기 렌더링 이후 크기가 변해도 충돌 상태로 방치되지 않아야 한다.

### 3) Human-Like Grouping

- 형제 노드는 기계적으로 일렬 정렬하기보다, 자연스럽게 덩어리로 읽혀야 한다.
- 가지 수가 많아도 한쪽으로 길게 늘어지는 형태를 줄여야 한다.
- 형제 수가 많아질수록 단일 수직열 대신 2차원적으로 퍼지는 배치 감각을 가져야 한다.
- 이 원칙은 시각적 그룹화에 관한 것이며, 실제 트리 깊이나 부모-자식 관계를 바꾸는 것을 의미하지 않는다.

### 4) Stable Layout

- 작은 변경은 가능한 한 국소적인 재배치로 끝나야 한다.
- 사용자가 기존 구조를 다시 학습해야 할 정도의 급격한 점프를 줄인다.

## 제안 방향

신규 레이아웃은 단순한 방향 전환이 아니라, "조밀 배치 + 비충돌 + 가지 응집"을 우선하는 별도 전략으로 정의합니다.

핵심 방향은 다음과 같습니다.

1. 서브트리 단위로 먼저 읽기 좋은 형태를 만들고,
2. 각 서브트리 사이 간격을 최소화하면서도 충돌은 막고,
3. 형제 노드가 많아질 때는 단일 수직 나열 대신 타일형/방사형에 가까운 확산 배치를 지향하고,
4. 큰 노드와 작은 노드가 섞여도 빈 공간이 과도하게 남지 않도록 배치합니다.

즉, 목표는 균일한 레이어 정렬이 아니라 "읽히는 밀도"를 만드는 것입니다.
또한 이 레이아웃은 기존 트리 topology를 재작성하지 않고, 좌표 배치만으로 품질을 개선하는 것을 전제로 합니다.

## 구현 전략

### 추천 전략 요약

신규 레이아웃은 여러 모드를 오가는 하이브리드보다, 하나의 일관된 배치 규칙을 끝까지 적용하는 방향이 더 적합합니다.

1. 전체 트리는 compact tree 계열을 기본 엔진으로 사용한다.
2. 모든 부모 노드는 자신의 자식 서브트리를 같은 형제 배치 규칙으로 합성한다.
3. 깊은 서브트리 때문에 생기는 과도한 공백은 contour 압축으로 줄인다.

즉, 기본은 "compact tree", 품질 개선 포인트는 "adaptive sibling placement"와 "subtree compression"입니다.

### 1) 기본 엔진: Compact Tree

가장 우선 추천하는 기본 엔진은 Reingold-Tilford / van der Ploeg 계열의 compact tree입니다.

- 가변 크기 노드에 강하다.
- 트리 구조를 유지하면서도 ELK layered보다 훨씬 촘촘하게 붙는다.
- 결과가 결정적이어서 같은 입력에서 같은 배치를 만들기 쉽다.
- 마인드맵처럼 부모-자식 관계가 분명한 구조에 적합하다.

이 레이아웃은 "현재 tree를 더 미세 조정하는 방식"보다, 처음부터 조밀 배치를 전제로 한다는 점에서 더 맞습니다.

### 2) 형제 배치 규칙: Adaptive Sibling Placement

형제 확산은 특정 시점에 별도 모드로 전환하는 것이 아니라, 부모가 자신의 자식 서브트리를 배치하는 매 단계에서 항상 적용되는 규칙이어야 합니다.

- 각 자식 서브트리를 먼저 재귀적으로 레이아웃한다.
- 부모는 자식 수, 서브트리 폭/높이, contour를 보고 형제들을 합성한다.
- 이 합성 규칙은 순수 수직 스택이 아니라, 필요에 따라 더 넓게 퍼지고 더 조밀하게 붙는 2차원 배치를 만든다.
- 중요한 점은 "형제 수가 N 이상이면 다른 알고리즘으로 바꾼다"가 아니라, 같은 규칙이 연속적으로 더 넓은 fan-out을 만든다는 것이다.

즉, 타일형, 부채형, 약한 방사형에 가까운 결과는 별도 레이아웃 모드가 아니라 이 형제 배치 규칙이 만들어내는 표현이어야 합니다.

### 3) 깊은 가지 압축: Contour / Profile Compression

이번 문제에서 가장 중요한 보완 기법 중 하나입니다.

- 각 서브트리의 bounding box만 비교하면, 깊은 가지 하나 때문에 형제 전체 간격이 커진다.
- 따라서 단순 박스 기준이 아니라, 서브트리의 실제 윤곽(contour/profile)을 비교해 최대한 가까이 밀착해야 한다.
- 깊은 가지 옆의 얕은 가지는 실제로 닿지 않는 한 더 가깝게 붙을 수 있어야 한다.

즉, "깊이가 깊다 = 형제 간격이 넓다"로 직접 연결되지 않도록 만드는 압축 규칙이 필요합니다.

### 4) 후처리 안정화: Measured Layout + Guarded Relayout

- 초기 레이아웃은 측정 가능한 크기로 먼저 계산한다.
- Markdown, 이미지, 임베드 등으로 실제 크기가 바뀌면 안전한 자동 재배치를 수행한다.
- 단, relayout은 debounce, threshold, retry limit을 둬서 무한 루프 없이 수렴해야 한다.

이 단계가 없으면 compact한 초기 배치를 해도 최종 렌더에서 다시 충돌이 생길 수 있습니다.

## 추천 기법

### 우선 적용할 기법

1. Compact Tree
   - 기본 트리 배치 엔진
   - 가변 크기 노드와 조밀 배치에 가장 적합
2. Contour/Profile 기반 서브트리 압축
   - 깊은 가지 때문에 생기는 세로/가로 공백 완화
   - 이번 문제 정의와 가장 직접적으로 연결됨
3. Adaptive Sibling Placement
   - 형제 수와 서브트리 크기에 따라 연속적으로 fan-out을 조정
   - 타일형/방사형에 가까운 사람식 배치를 하나의 규칙 안에서 구현
4. 실제 크기 측정 기반 재배치
   - 비동기 콘텐츠 이후 충돌 방지
5. Stable / Incremental Layout
   - 작은 변경 시 전체 점프 방지

### 보조적으로 고려할 기법

1. Quadrant / Sector 배치 패턴
   - 필요 시 형제 배치 규칙 안에서 참고할 수 있는 표현 패턴
2. Global Group Layout 유지
   - 다중 MindMap 간 전역 배치는 기존 방식 유지 가능
3. Density 파라미터
   - 사용자가 조밀도 수준을 조정할 수 있도록 할 여지가 있음

## 비추천 전략

### 1) ELK Layered 튜닝만으로 해결하려는 접근

- 방향성은 유지되지만, 형제 세로 확장과 깊은 서브트리 공백 문제를 근본적으로 해결하기 어렵다.
- "옵션 튜닝" 수준으로는 사람이 그린 듯한 밀도까지 가기 어렵다.

### 2) Treemap / Bin Packing 단독 적용

- 공간 효율은 좋아도 트리 구조와 엣지 읽기 흐름이 깨진다.
- 마인드맵의 의미 구조보다 사각형 채우기를 우선하게 된다.

### 3) Force-Directed 단독 적용

- 유기적인 배치는 가능하지만 결정성이 약하고, 계층 가독성이 떨어진다.
- 문서형 마인드맵보다 네트워크 그래프에 더 가깝다.

## 최종 추천안

v1에서는 아래 전략을 기본안으로 삼는 것이 가장 현실적입니다.

1. 기본 배치는 Compact Tree로 간다.
2. 모든 부모에서 동일한 Adaptive Sibling Placement 규칙으로 자식 서브트리를 합성한다.
3. 서브트리 간격 계산에는 Contour Compression을 적용한다.
4. 최종 렌더 이후 실제 크기 변화는 Guarded Relayout으로 수습한다.

즉, 신규 레이아웃은 "Compact Tree를 중심으로, 일관된 형제 배치 규칙과 깊은 가지 압축을 결합한 단일 레이아웃"으로 정의하는 것이 가장 적합합니다.

## v1 범위

### 포함

1. 신규 마인드맵 레이아웃 타입 1종 추가
2. 가변 크기 노드를 고려한 비충돌 배치
3. 형제/서브트리 간 간격을 줄이는 조밀 배치 규칙
4. 루트 기준 주요 가지가 자연스럽게 퍼지도록 하는 배치 규칙
5. 렌더 후 실제 크기 기준으로 충돌 여부를 다시 반영할 수 있는 안전한 재배치

### 제외

1. 자유곡선 드로잉처럼 완전히 수동 배치한 느낌의 랜덤 레이아웃
2. 일반 그래프 전체를 대상으로 하는 범용 오토레이아웃
3. 엣지 스타일 시스템 전면 개편
4. 수동 배치 편집 기능 자체의 확장
5. 레이아웃 단계에서 트리 구조를 자동으로 재작성하거나 가상 부모를 삽입하는 것

## 상세 요구사항

### 1) 비충돌 배치

- 레이아웃 결과에서 어떤 노드도 다른 노드와 겹치지 않아야 한다.
- 노드 크기는 fallback 크기가 아니라 실제 측정 크기를 우선 사용해야 한다.
- 초기 배치 후 크기 변화가 발생하면 재배치 또는 보정이 가능해야 한다.

### 2) 고밀도 배치

- 동일한 데이터셋에서 기존 `tree`, `bidirectional`보다 더 작은 bounding box를 목표로 한다.
- 서브트리 사이 dead space를 줄여 전체 구조가 한 덩어리로 읽히게 해야 한다.
- 자식 수가 많은 부모도 한 줄로 과도하게 늘어지지 않아야 한다.
- 특정 서브트리가 깊다고 해서 인접 형제 전체가 같은 폭의 공백을 강제로 떠안지 않아야 한다.

### 3) 사람 같은 읽기 흐름

- 가지별로 시선이 자연스럽게 이어져야 한다.
- 부모와 핵심 자식의 관계가 즉시 읽혀야 한다.
- 큰 노드가 있다고 해서 주변 전체가 과하게 벌어지지 않아야 한다.

### 4) 형제 확장 처리

- 형제 노드 수가 늘어나더라도 하나의 수직 열로만 계속 누적되지 않아야 한다.
- 형제 수, 서브트리 폭/높이, contour에 따라 배치 폭이 연속적으로 조정되어야 한다.
- 타일형 또는 방사형에 가까운 2차원 확산 배치는 별도 모드 전환이 아니라 동일 규칙의 결과로 나타나야 한다.
- 데이터 구조를 바꾸지 않은 상태에서도, 시각적으로 더 응집된 형제 배치를 만들어야 한다.

### 5) 깊은 서브트리 압축

- 한 형제의 자식 트리가 깊어져도 그 높이가 형제 전체 간격을 과도하게 밀어내지 않아야 한다.
- 깊은 가지 옆의 얕은 가지는 가능한 한 더 가깝게 붙어 배치될 수 있어야 한다.
- 수직 간격 계산이 곧바로 큰 수평 공백으로 번지지 않도록 서브트리 단위의 압축 규칙이 필요하다.

### 6) 안정성

- 입력 순서가 같으면 결과 좌표도 결정적이어야 한다.
- 일부 노드 추가/삭제 시 전체 구조가 불필요하게 크게 흔들리지 않아야 한다.
- 다수의 마인드맵 그룹이 있어도 각 그룹은 독립적으로 계산되어야 한다.

## 수용 기준

1. 대표 마인드맵 샘플에서 노드 박스 간 겹침이 0건이어야 한다.
2. 동일 샘플 기준 기존 `tree`, `bidirectional`보다 전체 bounding box 면적이 더 작아야 한다.
3. 자식 수가 많은 가지에서도 일렬로 과도하게 길어지는 현상이 완화되어야 한다.
4. 큰 노드와 작은 노드가 혼재한 샘플에서도 특정 가지만 비정상적으로 벌어지지 않아야 한다.
5. 형제 노드가 많은 샘플에서도 세로 길이만 과도하게 증가하지 않고, 2차원적으로 분산된 배치가 확인되어야 한다.
6. 깊은 서브트리와 얕은 서브트리가 함께 있는 샘플에서도 형제 사이 불필요한 세로/가로 공백이 크게 줄어들어야 한다.
7. 비동기 콘텐츠 로드 후에도 최종 상태에서 충돌 없이 안정된 레이아웃이 유지되어야 한다.
8. 사용자가 결과를 봤을 때 "자동 정렬된 트리"보다 "정리된 마인드맵"에 가깝다고 평가할 수 있어야 한다.

## 성공 판단 기준

### 정량 기준

1. 내부 기준 샘플셋에서 노드 충돌 재현율 0%
2. 동일 샘플셋에서 기존 주요 레이아웃 대비 bounding box 면적 감소
3. 재배치가 필요한 케이스에서도 무한 relayout loop 없이 수렴

### 정성 기준

1. 주요 가지가 덩어리로 읽힌다.
2. 전체 구조를 한 화면에서 더 빠르게 파악할 수 있다.
3. 형제 노드가 많은 구간에서도 세로로만 늘어지지 않고 자연스럽게 퍼진다.
4. 깊은 가지 하나 때문에 옆 가지 전체가 멀어지지 않고, 가지 간 밀도가 자연스럽게 유지된다.
5. 결과가 기계적 그리드보다 사람이 손으로 정돈한 마인드맵에 더 가깝다.

## 검증 메모

### 자동 검증 fixture

현재 dense layout 품질 비교는 아래 fixture를 기준으로 본다.

1. `mixed-size`
   - 큰 노드와 작은 노드가 섞인 기본 compact 품질
   - 반복 실행 결정성 검증
2. `multi-root`
   - 한 MindMap 안의 여러 루트 subtree와 root cluster compactness 검증
3. `sibling-heavy`
   - 형제가 많은 부모에서 수직열 붕괴가 줄어드는지 검증
4. `deep-vs-shallow`
   - 깊은 subtree 옆 얕은 subtree가 과도하게 멀어지지 않는지 검증

### 자동 검증 경로

1. `app/utils/strategies/compactPlacement.test.ts`
   - overlap 0, occupied bounds, vertical span, contour compression을 검증한다.
2. `app/hooks/useLayout.test.ts`
   - `compact` integration path에서 repeat-run determinism, sibling-heavy regression, placement frame metadata를 검증한다.
3. `app/components/GraphCanvas.test.tsx`
   - group-isolated relayout guard와 `120ms debounce`, `2px quantization`, `max 2 retries`를 검증한다.
4. `app/features/render/parseRenderGraph.test.ts`, `app/store/graph.test.ts`
   - `compact` surface와 기본 spacing `50`, multi-root metadata 전달을 검증한다.

### feature gate

이 기능의 구현 완료 판정은 repo 전체 `bun test`가 아니라, dense layout 관련 focused regression과 `bunx tsc --noEmit`를 기준으로 본다.
