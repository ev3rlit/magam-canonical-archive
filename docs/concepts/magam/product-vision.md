# 🚀 Product Vision & Architecture Wiki: Headless Agentic Canvas

## 1. 제품 비전 (Product Vision)

**"인간은 GUI로 창조하고, AI는 Headless로 확장하는 완벽한 멀티플레이어 워크스페이스"**

본 프로젝트는 단순한 시각화 도구가 아닙니다. 텍스트와 노드 기반의 데이터를 마인드맵/캔버스 형태로 렌더링하되, AI가 프로그래밍 코드를 짜듯 캔버스를 직관적으로 조작할 수 있도록 설계된 **데이터 주도형(Data-Driven) 헤드리스 캔버스 앱**입니다.

사용자가 이미 보유한 외부 AI 생태계(Claude Code, Cursor, ChatGPT 등)를 그대로 레버리지하는 **BYOC(Bring Your Own Client)** 철학을 따르며, 불필요한 SaaS 구독이나 복잡한 전역 설정(MCP) 없이 완벽한 Local-First 환경을 제공합니다.

---

## 2. 해결하고자 하는 문제 (The Problems We Solve)

- **GUI의 한계와 AI의 시각적 맹점:** 기존 캔버스 앱들은 인간의 마우스 드래그를 위해 설계되었습니다. AI는 픽셀 단위의 X, Y 좌표를 계산하는 데 취약(Hallucination)하며, 복잡한 캔버스를 편집하는 데 엄청난 토큰을 낭비합니다.
- **불필요한 추상화와 통합의 어려움:** 시중의 AI 캔버스 툴들은 자체적인 채팅창이나 무거운 프록시 서버를 앱 내부에 욱여넣어 시스템을 복잡하게 만듭니다.
- **멀티 세션과 파일 잠금(Lock)의 모순:** 로컬 파일 기반 DB(SQLite, 임베디드 Postgres)는 구조상 다중 프로세스의 동시 접근 시 Lock 충돌이 발생합니다. 이로 인해 GUI 앱과 백그라운드 AI 에이전트 간의 매끄러운 협업이 불가능했습니다.

## 3. 핵심 아키텍처: 'Houston' 동적 오케스트레이션 엔진

단일 진실의 원천(SSOT)을 유지하면서도 포트 충돌이나 데몬 관리의 복잡성을 없애기 위해, MSA의 리더 선출(Leader Election) 패턴을 차용한 **'Houston' 아키텍처**를 적용합니다.

- **동적 호스팅 (Dynamic Hosting):** GUI 앱이나 CLI 중 **누구든 먼저 실행되는 프로세스가 오케스트레이터(Host)**로 승격되어 PGlite(임베디드 DB)를 독점 마운트합니다.
- **gRPC over IPC:** 나중에 실행되는 프로세스는 호스트가 열어둔 로컬 소켓(Unix Domain Socket / Named Pipe)에 gRPC 클라이언트로 합류합니다. 네트워크 포트를 전혀 사용하지 않으므로 방화벽 이슈가 완벽히 차단됩니다.
- **완벽한 분리:** 서버/클라이언트의 역할을 유동적으로 스위칭하며, 모든 상태 변이와 통신은 Protobuf(Protocol Buffers)라는 강력한 타입 계약(Contract) 아래에서만 이루어집니다.

## 4. 기술 스택 (Tech Stack)

엔지니어링의 복잡도를 낮추고 실용적이고 깔끔한 코드를 유지하기 위해 다음의 기술을 채택합니다.

- **Core Logic & CLI:** `Node.js` (이벤트 드리븐 환경 및 뛰어난 파일/스트림 처리)
- **Database:** `PGlite` (WebAssembly로 구동되는 임베디드 PostgreSQL. 강력한 JSONB 쿼리와 Drizzle/Prisma 같은 ORM 생태계 100% 활용)
- **Communication:** `gRPC` (Protobuf 기반의 직렬화, HTTP/2 양방향 스트리밍을 통한 Sync Notify 구현)
- **View (Frontend):** `React / TSX` (오직 DB의 상태를 읽어 화면에 렌더링하는 'Dumb View' 역할 수행)
- **App Packaging:** `Tauri` 또는 `Electron` (운영체제 네이티브 API 및 IPC 소켓 접근용)

## 5. 킬러 피쳐 (Killer Features)

### ① High-Level Placement System (관계형 레이아웃 연산)

AI에게 픽셀 좌표(x: 120, y: 350)를 요구하지 않습니다. AI는 객체 간의 논리적 관계만 선언하며, 복잡한 충돌 방지 및 레이아웃 연산은 캔버스 렌더링 엔진이 담당합니다.

- **`anchor`:** 타겟 노드의 주위를 부유하듯 맴도는 독립적 객체 배치 (예: 파생 아이디어, 주석).
- **`attach`:** 부모 노드 내부에 완전히 귀속되어 함께 움직이는 메타데이터 (예: 태그, 상태 배지).
- **`group`:** 논리적 계층으로 묶어 배치.
- **`jitter`:** 그룹 내 오브젝트들의 transform(회전, 스케일)을 미세하게 랜덤 변형하여 아날로그적이고 자연스러운 UI 연출.

### ② 인터랙션 상태의 실시간 데이터화 (Contextual State Sync)

사용자가 캔버스에서 특정 노드들을 드래그하여 선택하면, 이 '현재 포커스 상태(editor_state)'가 즉시 DB에 저장됩니다.

- AI 에이전트(CLI)는 `get-active-context` 명령 하나로 현재 사용자의 작업 맥락을 즉시 파악합니다. 사용자가 데이터를 복사해서 AI에게 붙여넣을 필요가 없는 **Zero-Friction 경험**을 제공합니다.

### ③ Sync Notify via gRPC Server Streaming

DB 트랜잭션이 발생할 때마다 gRPC의 양방향 스트리밍을 통해 연결된 모든 클라이언트(GUI 뷰어)에게 델타(Delta) 이벤트가 브로드캐스트됩니다. AI가 터미널에서 명령을 내리면, 눈앞의 캔버스 앱 화면이 마법처럼 실시간으로 재구성됩니다.

## 6. AI 시대의 사용자 워크플로우 (The Agentic Workflow)

이 앱은 철저히 **기계(AI)와 인간의 협업**을 위해 디자인되었습니다.

1. **[인간의 직관]** 사용자는 캔버스 앱(GUI)을 열고, 마우스로 3개의 관련 아키텍처 노드를 선택합니다. (앱이 호스트 데몬으로 구동 중)
2. **[AI의 개입]** 사용자는 평소 쓰던 터미널(Claude Code)을 열고 자연어로 명령합니다. _"선택된 노드들을 분석해서, 각각의 장단점을 `attach` 모드로 태그 달아줘."_
3. **[Houston 통신]** 터미널의 `canvas-cli`는 gRPC 클라이언트로 GUI 앱의 IPC 소켓에 붙어, Protobuf 스키마에 맞춰 트랜잭션을 전송합니다.
4. **[데이터 처리]** GUI 앱 내의 PGlite가 JSONB 데이터를 업데이트합니다.
5. **[실시간 렌더링]** 업데이트 완료 즉시 캔버스 앱의 React 레이어가 변경된 DB 상태를 구독하여 화면에 태그들을 우아하게 애니메이션 렌더링합니다.

---

## 7. 향후 확장성 (Roadmap)

- **낙관적 동시성 제어 및 버전 관리:** DB 레코드에 버전을 부여하여, 인간과 AI가 동일 노드를 동시 타겟팅할 때 발생하는 충돌을 방어하고 완벽한 Undo/Redo 트레일 구축.
- **제안 모드 (Ghost UI):** AI가 대규모 레이아웃 변경을 시도할 때, 즉시 덮어쓰지 않고 반투명한 '초안' 상태로 화면에 띄워 사용자의 승인(Accept)을 받는 파이프라인.

이 설계는 복잡한 시스템의 불필요한 추상화를 철저히 배제하고, 데이터의 본질과 이벤트의 흐름에만 집중한 가장 현대적인 로컬 앱 아키텍처입니다.
